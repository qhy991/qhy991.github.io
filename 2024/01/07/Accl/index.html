Introduction

CNN加速器和transformer加速器不兼容的原因是，CNN加速器优化的是inner-product operation， 而不是matrix-matrix multiplication control flows。

现有的几个工作：

A^3 包含了几个估计策略来避免计算接近0的attention scores

SpAtten 利用cascade token pruning机制来根据低attention概率逐步修剪不重要的tokens，但是提出的 top-k 剪枝机制带来了很高的硬件开销。

Energon 使用一种混合精度多轮滤波算法来逼近top-k剪枝方法。但是只考虑到了attention 概率中的稀疏，没有考虑所用的乘法操作。

OPTIMUS 使用了一种集合关联重新排列压缩稀疏列 (SA-RCSC) 格式，以消除无效的乘法累加 (MAC) 操作。但是只考虑了weight。

为了提高硬件利用率，FFTRANS和SpAtten将matrix-matrix乘法分解成matrix-vector乘法操作，失去数据重用能力，也限制了并行度规模。

由于成本低廉，基于现场可编程门阵列（FPGA）的变压器加速器也被提出[20,21,22]。然而，由于位级可重构抽象和相应的高互连开销，它们的性能和功耗效率低下

Background and motivation

The transformer model

1）Compute Operation

2） Memory Requirements

两个模型的main memory的需求分别是52.8MB和3.4GB

Sparsity in Self-Attention

SpAtten 提出了一个top-k 的剪枝方法， 在runtime时，识别出产生最大的attention 概率的 query-key pairs。这种方法会带来一定的性能损失，而且还会带来很大的硬件开销。

Methodology

首先对transformer 模型进行weight-prune，要么使用Movement pruning，或者DynaTran。然后把网络映射成细粒的计算和存储操作。

DynaTran

没有使用top-k剪枝方法，这个工作使用了一种低硬件开销的动态推理方法，能够快速地将不关键的权重和激活值在runtime期间就去除。

通过判断数值的大小是否超过阈值。这种比较还能够并行进行，从而在一个周期内完成。

这里的阈值是在多个模型上实验得到的，在runtime时，通过查表计算即可。

The AccelTran Simulator

1）Tiling and datafflow

用的是tiled matrix multiplication

2）Accelerator Organization

使用的monolithic-3D integeration 来连接一个片上3D RRAM，也支持DRAM

首先从CPU获得指令数据流，然后从DRAM或RRAM读取 weights 和 embedding，这个mask buffer是和压缩方式有关。

3） Processing Elements

压缩后的数据分别存储在activation和weight FIFO

这个Post-compute saprsity module是将计算出来的数值再还原回应该有的二维格式保存在FIFO，最终存到Main memory

4）MAC Lanes

注意这里的数据位宽，初始都是IL+FL，然后变成2倍以防止溢出。

5） Dynamic Inference Modules

对于一个b x x x y的tile，使用 b x x x y个比较器。

6） Sparsity-aware Acceleration

使用binary-mask 框架来encode，使用一个和uncompressed的数据的shape一样的数组存储是否是NZ的信息。

通过对输入的两个mask进行bit-wise的AND 计算，两个XOR门就能参数filter mask。之后就只对需要的数据进行计算。

7）Simulator Flow

基于多个对Finfet，cache，main memory的模拟模型，使用python搭建了一个cycle-accurate的仿真器。

8）Smart scheduling of tiled operations

会存在计算、存储阻塞的情况

后面的不想看了，是口嗨了一下，这些东西在FPGA上或者CIM上实现不了。不过使用的模型，和任务可以参考一下。
